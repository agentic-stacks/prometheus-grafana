# Architecture

## Component Overview

The Prometheus-Grafana monitoring stack consists of five primary components. Each plays a distinct role in the collection, storage, alerting, and visualization of metrics.

### Prometheus Server

The central component that scrapes and stores time series data. The Prometheus server:

- **Discovers targets** through static configuration or service discovery mechanisms (Consul, Kubernetes, DNS, file-based, etc.).
- **Scrapes metrics** by pulling data from target HTTP endpoints at configured intervals.
- **Stores samples** in a local time series database (TSDB) on disk.
- **Evaluates rules** -- both recording rules (pre-computing expensive queries) and alerting rules (determining when conditions are met to fire alerts).
- **Serves PromQL queries** via its HTTP API, used by Grafana and other consumers.

Each Prometheus server operates autonomously. It does not depend on distributed storage or remote services to function, making individual instances highly reliable.

### Alertmanager

Handles alerts generated by Prometheus servers. The Alertmanager receives firing alerts from one or more Prometheus instances and applies three key functions before dispatching notifications:

- **Grouping** -- Classifies similar alerts into a single notification. For example, during a large outage affecting many instances, grouping combines hundreds of individual alerts into one consolidated notification. Grouping, timing, and receivers are configured via a routing tree in the Alertmanager configuration file.
- **Inhibition** -- Suppresses notifications for certain alerts when other specified alerts are already firing. For example, if an alert fires indicating an entire cluster is unreachable, Alertmanager can be configured to mute all other alerts concerning that cluster.
- **Silences** -- Mutes alerts for a given time period. Incoming alerts are checked against active silences using equality or regular expression matchers. Silences are configured in the Alertmanager web interface.

Alertmanager supports clustering for high availability. Prometheus should be pointed to a list of all Alertmanager instances rather than load-balancing traffic between them.

### Grafana

An open-source visualization and analytics platform that enables querying, visualizing, alerting on, and exploring metrics, logs, and traces. Within the Prometheus-Grafana stack, Grafana serves as the presentation layer:

- **Dashboards** -- Provides 25+ visualization panel types (time series, gauges, heatmaps, tables, bar charts, etc.) organized into dashboards with template variables for reusable designs.
- **Query editor** -- Offers a dedicated PromQL query editor for constructing and testing metric queries against Prometheus data sources.
- **Alerting** -- Supports Grafana-managed alert rules and data-source-managed rules, with notification routing to PagerDuty, Slack, email, and other channels.
- **Exploration** -- Includes an Explore interface for ad-hoc investigation of metrics without requiring a pre-built dashboard.
- **Access control** -- Provides role-based access control, folder and dashboard-level permissions, and data-source-level access restrictions.

Grafana has native built-in support for Prometheus as a data source -- no additional plugins are required.

### Exporters

Specialized adapters that expose metrics from third-party systems in a format Prometheus can scrape. Exporters bridge systems that do not natively expose Prometheus-format metrics. Common exporters include:

| Exporter | Purpose |
|---|---|
| `node_exporter` | Hardware and OS metrics (CPU, memory, disk, network) for Linux/Unix hosts. |
| `blackbox_exporter` | Probing of endpoints over HTTP, HTTPS, DNS, TCP, ICMP, and gRPC. |
| `mysqld_exporter` | MySQL server metrics. |
| `snmp_exporter` | Metrics from SNMP-enabled devices. |

Exporters run as separate processes alongside the systems they instrument and expose an HTTP `/metrics` endpoint that Prometheus scrapes.

### Pushgateway

An intermediary service for short-lived jobs that may terminate before Prometheus can scrape them. These jobs push their metrics to the Pushgateway, and Prometheus then scrapes the Pushgateway as a regular target.

**When to use:** Batch jobs, cron jobs, or other ephemeral processes that cannot expose a persistent HTTP endpoint for scraping.

**When not to use:** The Pushgateway is not intended as an aggregation proxy or as a replacement for the pull model for long-running services. Using it outside its intended scope can mask instance-level visibility and introduce stale metric data.

## Data Flow

### Metrics Collection and Visualization

```
Instrumented       Prometheus        Local         Grafana
Applications  -->   Server     -->   TSDB    -->  Dashboards
& Exporters       (scrape)        (storage)     (PromQL queries)
```

1. **Scrape** -- Prometheus discovers targets via static configuration or service discovery and pulls metrics from each target's `/metrics` HTTP endpoint at a configured scrape interval (default: every 60 seconds in recent versions, commonly set to 15s or 30s).
2. **Store in TSDB** -- Scraped samples are written to the local time series database. The TSDB stores data in two-hour blocks on disk, with a write-ahead log (WAL) for durability. An in-memory head block handles the most recent data.
3. **Query with PromQL** -- Grafana (or other consumers) issues PromQL queries to the Prometheus HTTP API. PromQL provides filtering by labels, aggregation across dimensions, and functions such as `rate()`, `histogram_quantile()`, and arithmetic operations.
4. **Visualize** -- Grafana renders query results using the configured panel type (time series graph, gauge, table, heatmap, etc.) on dashboards.

### Alerting Pipeline

```
Prometheus        Alertmanager       Notification
  Server     -->   (grouping,   -->   Channels
(rule eval)       inhibition,       (email, Slack,
                  silencing)         PagerDuty, etc.)
```

1. **Rule evaluation** -- Prometheus evaluates alerting rules at a configured interval. When an alerting rule condition is met, Prometheus generates an alert.
2. **Alert dispatch** -- Prometheus sends firing (and resolved) alerts to the Alertmanager via its HTTP API.
3. **Alert processing** -- Alertmanager groups related alerts, checks inhibition rules, matches against active silences, and routes alerts through its routing tree to the appropriate receiver.
4. **Notification** -- Alertmanager dispatches notifications to the configured channels (email, Slack, PagerDuty, OpsGenie, webhooks, etc.).

## Network Requirements

| Component | Default Port | Protocol | Purpose |
|---|---|---|---|
| Prometheus server | 9090 | HTTP/TCP | Web UI, HTTP API (PromQL queries), federation endpoint, remote write receiver |
| Grafana | 3000 | HTTP/TCP | Web UI, dashboard rendering, data source proxy, HTTP API |
| Alertmanager | 9093 | HTTP/TCP | Web UI, HTTP API (alert receiver, silence management) |
| Alertmanager | 9094 | TCP/UDP | Cluster gossip protocol for high-availability peer communication |
| node_exporter | 9100 | HTTP/TCP | Host metrics endpoint (`/metrics`) |
| Pushgateway | 9091 | HTTP/TCP | Push endpoint for short-lived job metrics, web UI |

All scraping communication flows from Prometheus to targets (pull model). Prometheus initiates outbound HTTP connections to each target's port. Alertmanager receives inbound HTTP connections from Prometheus on port 9093 and communicates with peers on port 9094 for cluster state synchronization.

Grafana initiates outbound HTTP connections to Prometheus on port 9090 to execute PromQL queries. Grafana does not require direct network access to exporters or Alertmanager for basic dashboard functionality.

## Deployment Patterns

### Single Server

The simplest deployment runs Prometheus, Alertmanager, and Grafana on a single host. This pattern works well for:

- Small to medium environments (up to thousands of targets).
- Development and testing setups.
- Teams getting started with monitoring.

Each Prometheus server operates autonomously with its own local TSDB. This independence from external dependencies makes standalone instances highly reliable.

### Distributed (Federation and Remote Storage)

For environments that exceed the capacity of a single Prometheus server, two scaling approaches are available:

**Hierarchical federation** allows Prometheus to scale to environments with tens of data centers and millions of nodes. Local per-datacenter Prometheus servers collect detailed instance-level metrics. Higher-level global servers scrape aggregated job-level data from the local servers via the `/federate` endpoint. This creates both granular local visibility and high-level global insights.

**Cross-service federation** enables one service's Prometheus instance to pull selected metrics from another service's Prometheus server. This allows unified alerting and querying across both datasets within a single server -- for example, pulling cluster resource metrics (CPU, memory) from a scheduler's Prometheus into a service-specific instance.

**Remote storage** integrations (via the `remote_write` and `remote_read` APIs) allow Prometheus to send data to or read data from external long-term storage systems such as Thanos, Cortex/Mimir, VictoriaMetrics, or InfluxDB. This extends retention beyond local disk capacity and enables a global query view across multiple Prometheus instances.

### Kubernetes

Kubernetes is the most common deployment target for the Prometheus-Grafana stack. Three primary approaches exist:

**Helm charts** -- The `kube-prometheus-stack` Helm chart bundles Prometheus, Alertmanager, Grafana, node_exporter, kube-state-metrics, and default dashboards/alerting rules into a single installable package. This is the fastest path to a production-ready monitoring stack on Kubernetes.

**Prometheus Operator** -- A Kubernetes operator that manages Prometheus and Alertmanager instances via custom resources (`Prometheus`, `Alertmanager`, `ServiceMonitor`, `PodMonitor`, `PrometheusRule`). The operator handles configuration, scaling, and lifecycle management. The `kube-prometheus-stack` Helm chart deploys the operator under the hood.

**Service discovery** -- Prometheus has native Kubernetes service discovery (`kubernetes_sd_configs`) that automatically discovers pods, services, endpoints, nodes, and ingresses as scrape targets. Labels from Kubernetes objects (annotations, labels, namespaces) are available for relabeling, enabling convention-based scrape configuration such as annotating pods with `prometheus.io/scrape: "true"`.

## How Grafana Connects to Prometheus

### Data Source Configuration

Grafana connects to Prometheus by configuring it as a data source. Prometheus support is built into Grafana natively -- no plugin installation is required. Configuration requires specifying:

- **URL** -- The HTTP address of the Prometheus server (e.g., `http://prometheus:9090` or `http://localhost:9090`).
- **Access mode** -- Grafana supports a server (proxy) access mode where the Grafana backend proxies requests to Prometheus, and a browser (direct) mode where the browser connects directly.
- **Authentication** -- Optional settings for basic auth, TLS client certificates, or cloud-specific authentication (AWS SigV4 for Amazon Managed Prometheus, Azure AD for Azure Monitor).

Data sources can be configured through the Grafana web UI or provisioned declaratively via YAML files in the `provisioning/datasources/` directory for infrastructure-as-code workflows.

### Query Proxy

When a user views a dashboard or runs a query in the Explore interface, Grafana acts as a proxy between the user's browser and the Prometheus server:

1. The browser sends the query request to the Grafana backend.
2. Grafana's backend translates this into an HTTP API call to the Prometheus server's `/api/v1/query` (instant) or `/api/v1/query_range` (range) endpoint.
3. Prometheus evaluates the PromQL expression against its TSDB and returns the result set as JSON.
4. Grafana's backend forwards the response to the browser, which renders the configured visualization.

This proxy architecture keeps the Prometheus server shielded from direct browser access, allows Grafana to enforce its own authentication and authorization layer, and enables Grafana to work with any backend that implements the Prometheus querying API (including Thanos, Cortex/Mimir, and VictoriaMetrics).
